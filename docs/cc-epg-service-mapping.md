# `cc-epg-service` Paper Mapping

## Purpose

This document maps the current `cc-epg-service` Kustomize setup to the current
`App` contract without doing a live migration yet.

It is the first paper migration template for Wave 1.

## Current Resource Shape

Current Kustomize resources:

- one `Deployment`
- one `Service`
- one ingress object per environment
- one `ExternalSecret`
- one generated config map
- one production `PodDisruptionBudget`

## Current Runtime Contract

The deployment imports environment from:

- generated config map `cc-epg-service`
- generated secret `cc-epg-service`
- existing secret `newrelic`

The app code requires these runtime variables:

- `AUTH_AUDIENCE`
- `AUTH_INHOSPITAL_API_GATEWAY`
- `AUTH_ISSUER`
- `AUTH_JWKS_URI`
- `AWS_S3_ACCESS_KEY_ID`
- `AWS_S3_SECRET_ACCESS_KEY`
- `AWS_S3_REGION`
- `AWS_S3_BUCKET_NAME`
- `LOG_LEVEL`
- `NODE_ENV`
- `SERVER_PORT`
- `CRON_JOB_CACHE_UPDATE`

Optional by code:

- `HTTP_LOGGING`

## Direct Mapping To `App`

Deployment basics:

- image -> `spec.image`
- replicas -> `spec.replicas`
- resources -> `spec.runtime.resources`
- image pull secret `regcred` -> `spec.runtime.imagePullSecrets`
- service account -> `spec.runtime.serviceAccountName`
- node selector -> `spec.runtime.nodeSelector`
- topology spread -> `spec.runtime.topologySpread`
- readiness probe `/v1/health` -> `spec.runtime.readinessProbe`
- liveness probe `/v1/health` -> `spec.runtime.livenessProbe`

Service:

- service port `8000` -> `spec.port`
- service type -> `spec.service.type`

Config map literals:

- `LOG_LEVEL`
- `NODE_ENV`
- `SERVER_PORT`
- `AWS_S3_REGION`
- `AWS_S3_BUCKET_NAME`
- `CRON_JOB_CACHE_UPDATE`
- per-environment auth values

These are good fits for:

- `spec.config.data`

Secret imports:

- `cc-epg-service` secret generated by `ExternalSecret`
- `newrelic` secret imported via `envFrom`

These are good fits for:

- `spec.externalSecret.*` for the generated app secret
- `spec.config.envFromSecrets` for the target app secret and `newrelic`

Availability and observability:

- production PDB -> `spec.disruptionBudget.*`
- service monitor intent -> `spec.serviceMonitor.*`

Exposure:

- ingress -> `spec.route.*` with `HTTPRoute`

## What Still Needs A Decision

The current repo mounts:

- `emptyDir` at `/opt/npm-cache`

That is not expressible in the current `App` storage model.

Options:

1. decide that the mount is not required for the first migration spike
2. keep a small raw patch beside `App`
3. expand `App` later to support `emptyDir`

For a first spike, option `1` or `2` is cleaner than stretching storage now.

## Recommended First `App` Shape

For a paper migration, the first target shape should be:

- one `App`
- one generated secret via `externalSecret`
- one imported `newrelic` secret via `config.envFromSecrets`
- `HTTPRoute` instead of ingress
- no attempt to model `emptyDir` yet

## Gaps After Paper Mapping

Remaining gap for a clean migration:

- explicit decision on the `emptyDir` npm cache mount

Everything else is already within the current validated `App` scope or close
enough for a controlled migration spike.
